# 全局工作约定 — AI 协作的文档规范

> **版本** v1.1(2026-04-30 修订)
> **作用域**:本文件被 Claude Code 自动加载到**每个 session**,不论项目。
> **目标**:让任何一个新 session(同一个 AI 不同次启动 / 不同 AI / 不同人)在 30 秒内回答两个问题——"现在到哪一步"、"下一步该做什么"——而不依赖会话历史。
>
> **修改本文件**:任何条款的修订必须在底部"修订日志"加一行(日期 + 一句话动机)。本文件本身也是有可能滞后的真相源,session 接手时若发现规则与项目现状冲突,**优先信项目里的 CLAUDE.md**,然后告知用户哪条规则该升级。

本文档使用 RFC 2119 风格的强度标记:

- **[MUST]** — 不遵守视为破坏约定
- **[SHOULD]** — 默认遵守,有充分理由可以偏离(偏离时在项目 CLAUDE.md 写明原因)
- **[MAY]** — 可选模式,适配项目特点

---

## 1. 核心原则

### 1.1 [MUST] 文档是真相源(canonical source of truth)

代码注释、根目录 `README.md`、git log message **都可能滞后**。AI 接手后做任何架构判断之前,必须以项目的设计文档为准。

具体落地:

- 项目根 **必须** 有一个固定入口文件(默认 `CLAUDE.md`),在第一段明确声明"本项目的真相源是 `<docs/路径>`"
- 文档与代码不一致时,要么改代码、要么改文档,**不允许**让两者继续矛盾

### 1.2 [MUST] 接手 30 秒原则

任何一份规划类文档(roadmap / milestones / todo)开头必须能让一个**完全没有上下文**的接手者在 30 秒内回答:

1. 现在到哪一步?
2. 下一步做什么?

如果一份规划文档读完一遍还无法回答这两个问题,说明它已经失效或写偏了。

### 1.3 [MUST] 架构不变量清单

每个项目的 `CLAUDE.md` 必须列出**绝对不能破坏**的几条架构约束(通常 3–6 条)。例:跨平台编译标志、模块边界、依赖方向、安全约束。

AI 在动手前 **必须** 先扫一眼这份清单,改动可能影响某条不变量时主动汇报。

### 1.4 [SHOULD] 产品 / 程序设计分离

每个独立功能在 `docs/tabs/<feature>/` (或等价目录)下放两份文档:

- `product.md` — 产品意图(目标、非目标、UX、字段语义、边界规则)
- `program.md` — 实现细节(代码组织、API 契约、命令构建、测试)

理由:产品意图变化频率与实现变化频率不同,放一起容易混淆"为什么这样"和"怎么这样"。

### 1.5 [SHOULD] 三档规划文档(规模适配,见 §6)

详见 §2、§3、§4、§5。

### 1.6 [SHOULD] AI 友好的命名与索引

- 文档**必须有索引**:`docs/README.md` 列出所有文档及其用途,新文档加进来必须同步到索引
- 链接**优先用相对路径**(不依赖工作目录)
- 文件名**用语义而非顺序**(`timeline-design.md` 而非 `01-design.md`,前者迁移时不会因为顺序变了一窝错乱)

## 2. 三档规划文档体系

每个**有持续开发的项目**都应该有这三份(规模可裁剪,见 §6):

| 文档 | 粒度 | 回答的问题 | 更新频率 |
|------|------|-----------|---------|
| `roadmap.md` | **粗** — 功能级 | 接下来要做哪些功能?边界(非目标)在哪? | 月级 |
| `milestones.md` (+ `milestones/`) | **中** — 单功能里程碑 | 当前在做哪个功能?到第几个 M? | 周级 |
| `todo.md` (+ `todo/`) | **细** — 当前 M 的具体动作 | 这个 M 还差哪几步? | 日级 |

### 2.1 文档职责清晰边界

- `roadmap.md` **只**记录功能与版本的粗粒度信息;不记 commit、不记代码细节
- `milestones.md` 主文件 **只**做索引(进行中 + 归档),具体 M 详情在 per-branch 文件里(见 §3)
- `todo/<branch>.md` 服务当前 active 分支;分支结束、文件删除

任何一份文档"知道得太多"都意味着职责越界——通常是没及时晋升到对应档位或没及时归档。

### 2.2 晋升触发条件(三档之间的内容流转)

| 触发 | 动作 |
|------|------|
| 某功能正式启动开发 | `roadmap.md` 那行标"⏳ 进行中,见 milestones.md";`milestones.md` 索引加一行;创建对应分支 + `docs/milestones/<branch>.md`(分支 → 文件名见 §5) |
| 开始一个具体 M | 把那个 M 的交付内容拆成可勾选清单,**整段填入** `docs/todo/<branch>.md`;`docs/milestones/<branch>.md` 那行从 ⏳ 改 🚧 |
| M 完结 | `docs/milestones/<branch>.md` 那行标 ✅ + commit 哈希 + 日期;`docs/todo/<branch>.md` **整段清空**(只保留模板注释或开下一个 M) |
| 整个功能完结 | `docs/milestones/<branch>.md` **`git mv` 到** `docs/milestones/archive/`;主索引 `milestones.md` 中"进行中"那行挪到"已归档索引";`docs/todo/<branch>.md` 删除;`roadmap.md` 在"已发布版本"加一行 |

**[MUST]** 项目的 `CLAUDE.md` 或 `docs/README.md` 必须把这张表抄一份(让接手者不用看本文件也能知道流转规则)。

## 3. 归档策略

### 3.1 [MUST] milestones 主文件保持精简

`docs/milestones.md` **只**保留三段:

1. 简介 + 三档规则参考
2. **进行中**索引:列出当前 active 的分支与对应 milestone 文件链接(每行 ≤120 字符)
3. **已归档**索引:每个完成的功能一行,链接到 `archive/` 路径

具体的 M 详情、commit、归档段落 **必须**在独立的 per-branch 文件里(见 §5),不在主文件。这样:

- AI 加载 `milestones.md` 的成本恒定(只看索引)
- 历史归档不会被新迁移项目挤掉
- per-branch 修改不污染主文件,合并冲突极少

### 3.2 [MUST] 完成的功能 git mv 到 archive/

某个分支整体完工时:

1. 该 feature 的 `docs/milestones/<branch>.md` **不要**复制到归档,**直接 `git mv`** 到 `docs/milestones/archive/<branch>.md`(保留 git 历史血脉,git rename detection 自动连起来)
2. `docs/milestones.md` 主文件:从"进行中索引"剪到"已归档索引",链接更新到 archive 路径
3. `docs/todo/<branch>.md` 删除(M 完结清空原则,见 §2.2)

### 3.3 [SHOULD] roadmap 历史压缩

`roadmap.md` 的"已发布版本"区:

- 保持单文件,因为粗粒度行数少
- **5 个版本一压缩**:把 0.1.x–0.4.x 折叠为一行 `v0.x — 早期阶段(详见 archive/early-versions.md)`,详情挪到 `docs/roadmap/archive/early-versions.md`

## 4. 多人协作扩展

### 4.1 [SHOULD] per-branch 文件天然解耦

每个 active feature/branch **独占**一份 `docs/milestones/<branch>.md`,其他人**不动它**。多分支并行开发时:

- Alice 在 `multitrack` 分支只改 `docs/milestones/multitrack.md`
- Bob 在 `auth-rewrite` 分支只改 `docs/milestones/auth-rewrite.md`
- 两人的 PR 自然不冲突(除非都要改主索引,这种情况下走 §4.2)

### 4.2 [MUST] 共享文件改动走 PR review

只有以下场景需要改"共享文件"(`roadmap.md` / `docs/milestones.md` 主索引 / `docs/README.md`):

- 启动新功能(roadmap 加一行 + milestones 索引加一行)
- 完工归档(milestones 索引中"进行中"挪到"已归档")
- 大方向变更(roadmap 调整非目标 / 删功能)

这些改动 **必须** 走 PR review,不可绕过 main。

### 4.3 [SHOULD] Markdown 友好合并

无论几人协作:

- 每行写一件事,不在一行用 `;` / `,` 拼多句 — 减少 line-level 冲突
- 表格新行**追加到顶部**(进行中 / 已归档区都按时间倒序),最新的在最上面,git diff 干净,且接手者第一眼看到最近的
- 时间用绝对日期 `YYYY-MM-DD`,不用相对(`Thursday` / `上周三`)— 异步协作时相对时间会指代不明

### 4.4 [MAY] CODEOWNERS 锁文件

中大型项目可在 `.github/CODEOWNERS` 锁定:

```text
docs/milestones/multitrack.md      @alice
docs/milestones/auth-rewrite.md    @bob
docs/milestones.md                 @maintainers   # 共享索引
docs/roadmap.md                    @maintainers   # 大方向
```

## 5. 分支 → 文件的发现协议

### 5.1 [MUST] 文件路径机械推导

```text
docs/milestones/<branch_name with "/" → "_">.md
docs/todo/<branch_name with "/" → "_">.md
```

| 分支 | milestones 文件 | todo 文件 |
|------|-----------------|-----------|
| `multitrack` | `docs/milestones/multitrack.md` | `docs/todo/multitrack.md` |
| `feature-5.3.2/zhouyijin/eating-fish` | `docs/milestones/feature-5.3.2_zhouyijin_eating-fish.md` | `docs/todo/feature-5.3.2_zhouyijin_eating-fish.md` |
| `v0.5.0` | `docs/milestones/v0.5.0.md` | `docs/todo/v0.5.0.md` |
| `bugfix/login-crash` | `docs/milestones/bugfix_login-crash.md` | `docs/todo/bugfix_login-crash.md` |

**唯一允许的字符变换是 `/` → `_`**(文件系统兼容)。其他不变(包括大小写、连字符、点)。

### 5.2 [MUST] AI 接手 session 的发现序列

每个 session 第一步:

```text
1. git branch --show-current   → 拿到当前分支名(如 "multitrack")
2. 推导 docs/milestones/<derived>.md  → 打开,看到当前 milestone 上下文
3. 推导 docs/todo/<derived>.md       → 打开,看到具体待办
4. 任一文件不存在 → 主动问用户:
   "当前分支 <X> 没有对应的 milestone 文件,你想:
    (a) 新建?
    (b) 这是探索性工作,不需要 milestone 跟踪?
    (c) 用其他文件名?"
5. 分支是 main / develop / hotfix-* 等非 feature 分支 → 不进 milestone 上下文,主动问用户做什么
```

### 5.3 [MUST] 项目 CLAUDE.md 必须抄一份本协议

让接手者不读 `~/.claude/CLAUDE.md` 也能知道怎么定位文件。同时项目 CLAUDE.md 可以**override**这条规则(比如某些项目用 `/` 分隔的 PR 文件名),但 override 必须显式写出来,不能默认偏离。

## 6. 项目规模适配

不是每个项目都需要全套体系。按代码量分级:

### 6.1 [MAY] 微项目(< 500 行,单文件脚本 / 一次性工具)

- 跳过 `docs/`,只在源文件顶部写 `# 设计意图` 注释块
- `CLAUDE.md` 可省略;若有,只写"这是什么"+ 一两条不变量

### 6.2 [SHOULD] 小项目(500–5000 行,单一职责库 / 小工具)

- 简化为**两档**:`roadmap.md` + `todo.md`(把 milestones 合进 roadmap)
- 不分 product/program,功能设计写在一份 `design.md`
- §3 归档 / §4 多人协作可暂缓,直到团队人数 > 1 或并行迁移 > 1 时再启用
- `CLAUDE.md` 必须有,照模板(见 §7)裁剪

### 6.3 [SHOULD] 中大型项目(5000+ 行,多模块产品)

- 完整三档:`roadmap.md` + `milestones.md` 索引 + `milestones/` per-branch + `todo/` per-branch
- §3 归档、§4 多人协作、§5 发现协议 全部启用
- 按功能子目录分 product/program
- 共享层独立目录(`core/` / `shared/` 等)
- 完整 `CLAUDE.md` + `docs/README.md` 双索引

### 6.4 升级触发

- 小升中:milestones 主文件 > 500 行,或同时有 ≥ 2 个 active 迁移
- 中升大:多人协作 / 团队 > 3 人

**当前在做的 M 完结后**升级,不要在 M 中途换体系。

## 7. 项目 CLAUDE.md 的最小骨架

每个项目根的 `CLAUDE.md` **必须**包含以下几节(顺序固定):

```markdown
# CLAUDE.md

> 给未来的 Claude:**先读 [docs/](docs/) 再动手**。本仓库的设计文档是真相源。
> 本项目遵循 ~/.claude/CLAUDE.md v1.1 的文档规范。

## 第零步:看当前进度

[**必须抄**:分支 → 文件的发现协议,见 ~/.claude/CLAUDE.md §5]

执行序列:
1. `git branch --show-current` 拿到分支名
2. `docs/milestones/<branch with / → _>.md` 当前里程碑
3. `docs/todo/<branch with / → _>.md` 当前 M 的待办
4. 文件不存在或在 main → 主动问用户

## 第一步:读设计文档

[指向 docs/README.md 或同级文档索引]

## 项目一句话定位

[≤ 3 句话:这是什么 + 关键架构选择]

## 不可违反的架构不变量

[3–6 条 [MUST] 级约束,带简短 why]

## 关键目录速查

[列主要源代码目录及职责]

## 测试 / 健康检查

[一段 bash 命令块,跑完能验证项目是健康的]

## 沟通约定(可选)

[语言偏好、commit message 风格等]
```

**[SHOULD]** 项目首次引入本规范时,在 CLAUDE.md 顶部写一行 `本项目遵循 ~/.claude/CLAUDE.md v1.x`,出问题时方便溯源。

## 8. 反模式(已识别的常见错误)

| 反模式 | 为什么不好 | 正确做法 |
|--------|-----------|---------|
| 把"路线图 + 技术债 + 历史版本"塞一份 `roadmap.md` | 三类信息更新频率不同,合在一起谁也看不清 | 拆开,或至少明确分小节 |
| 用 commit log 当 milestones | 接手者必须 `git log` 才能知道进度,不符合 30 秒原则 | milestones.md 主动维护,commit 只在归档行里链接 |
| 文档间用绝对路径 `[link](C:\...)` | 换机器/换平台立刻全断 | 全用相对路径 |
| 每个新功能起一份独立文档但不更新索引 | 索引滞后等于文档不存在 | 加文档时**同一个 PR/commit** 更新索引 |
| 把"暂时这么实现"写进 `program.md` | 临时实现变长期债务,文档变诱导 | 临时方案放 todo.md 或单独的 `tech-debt.md`,完工后回写正式文档 |
| AI 修改设计文档时不告诉用户 | 用户失去对真相源的所有权 | 修改文档前明确告知"我打算改 docs/X 的 Y 节,因为 Z" |
| 不同 feature 的 milestone 内容混到同一份文件 | 多人改时合并冲突;接手者读不出"现在做哪个" | 按 §5 拆成 per-branch 文件,主索引只列链接 |
| 在 main / hotfix 分支上动 milestone 文件 | 当前分支不在 milestone 上下文 | 按 §5.2 第 5 条,主动问用户,不擅自动 |
| 已完工的 feature 内容不归档,主 milestones.md 越来越长 | 加载成本线性增长,接手者扫读痛苦 | 按 §3.2 `git mv` 到 archive/,主索引只留一行 |

## 9. 与 Claude Code 系统的衔接

- **Memory 系统**(`~/.claude/projects/<proj>/memory/`):放跨 session 的**事实记忆**(用户偏好、项目历史决策),不放规则。规则属于本文件。
- **TodoWrite 工具**:session 内的任务追踪,与 `docs/todo/<branch>.md` 互补——前者是会话级(关掉就消失),后者是项目级(跨 session 持久)。当前 M 的实际可勾选清单**必须**也写到 `docs/todo/<branch>.md`,TodoWrite 只是 session 内的影分身。
- **Plan 模式**:开始一个 M 前若需要拆解,可以走 plan;plan 通过后**必须**把结果落到 `docs/todo/<branch>.md`,不能只留在 plan 状态。
