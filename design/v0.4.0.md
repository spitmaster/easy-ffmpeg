# v0.4.0 — 双产物（Web + Wails 桌面）可行性分析

> 本文档评估把 Easy FFmpeg 从"纯 Web（Jupyter 式）"升级为"Web + Wails 桌面"双产物的可行性、成本与推荐路径。
>
> **结论先行**：成本属于**中低**——前提是采用"共享 HTTP 后端 + Wails 仅充当 WebView 外壳"的方案，不重写前端、不重写后端，只新增一个并列入口。
>
> 本版本与 [overview.md §6 非目标](overview.md) 的"不打包 webview（不走 Electron / Tauri / Wails 路线）"形成正面冲突，详见 §8。

---

## 1. 目标

发版后输出**两个**可分发产物，共用一份代码：

| 产物 | 形态 | 等价类比 |
|------|------|----------|
| **Web 版**（保留现状） | 命令行启动一个本地 HTTP 服务，调用系统浏览器打开 UI | Jupyter Notebook |
| **桌面版**（Wails） | 双击启动一个独立窗口，内置 WebView 直接渲染 UI | Electron / Tauri 应用 |

非目标（v0.4.0 不做）：

- 移动端 / 平板专门适配
- 完全切换到 Wails 原生 binding（`runtime.EventsEmit` / `Bind`）取代 HTTP+SSE —— 见 §3 方案 B 拒绝理由
- 把剪辑器 / 音频两个模块拆成独立的 Wails 子窗口
- 系统托盘、自动更新、原生菜单等桌面端"高级体验"

---

## 2. 当前架构对 Wails 改造的友好度评估

逐项打分（✅ 友好 / ⚠️ 需调整 / ❌ 阻塞）：

| 点位 | 现状 | 评估 |
|------|------|------|
| 后端模块化 | `server/` 是单一 `New() *Server` + `routes(mux)`，无全局状态 | ✅ 可整体复用 |
| 前后端通信 | HTTP REST + 一条共享 SSE（`/api/convert/stream`） | ✅ Wails WebView 通过 `http://127.0.0.1:port/` 同样能跑；SSE 也走得通 |
| 前端构建 | 原生 HTML/CSS/JS，零构建步骤，`go:embed` 打包 | ✅ Wails 完全不挑前端栈 |
| 模块边界 | `editor/` 已按 [editor-module-design.md §7](editor-module-design.md) 设计为可独立编译 | ✅ 同样的 wiring 模式可以套用到 Wails 入口 |
| FFmpeg 子进程 | `os/exec` + `procutil.HideWindow`（CREATE_NO_WINDOW） | ✅ Wails 下行为不变 |
| FFmpeg 嵌入 | 7z + `go:embed` + `~/.easy-ffmpeg/bin-<hash>/` | ✅ 和 Wails 无关，原样复用 |
| 文件选择 | 自绘 HTML 文件浏览器（`/api/fs/list` + `/api/fs/home`） | ✅ 可保留；可选地接入 Wails 原生 dialog（增量改进，非必需） |
| 浏览器拉起 | `internal/browser/open.go` | ⚠️ Wails 模式下不需要，要按编译模式分支 |
| 退出按钮 | `POST /api/quit` → `srv.RequestShutdown()` | ⚠️ Wails 模式下应改为关窗触发（保留 HTTP endpoint 也无害） |
| 编译方式 | `CGO_ENABLED=0` 纯 Go 静态链接，可从 Windows 跨编 4 平台 | ❌ Wails 强制 `CGO_ENABLED=1`，需要本机 C 工具链；不能再"一台机器编全平台" |
| 运行依赖 | 无 | ⚠️ Wails 桌面版的 Linux 需要 WebKitGTK；Windows 需要 WebView2 Runtime（Win10+ 通常预装，旧机器需引导安装） |
| 文档"非目标" | overview.md §6 / roadmap.md §6 明文写"不走 Wails 路线" | ⚠️ 本迭代会撤销这一条，需要同步改文档 |

**总评**：现有架构对"加一层 Wails 外壳"是高度友好的，因为：

1. UI 完全是 Web 标准，Wails 只要把 WebView 指过去即可；
2. `editor/` 已经做过一次"为可独立编译而设计"的训练，知道如何在 `cmd/` 下加新入口；
3. FFmpeg 嵌入、SSE 进度、剪辑器全部都是与"宿主是浏览器还是 WebView"无关的逻辑。

唯一真实的工程税在 §6（CGO 工具链 + 平台 runtime）。

## 3. 改造方案对比

### 方案 A：共享 HTTP 后端，Wails 仅充当 WebView 外壳（推荐）

桌面版的 Wails 入口启动**和 Web 版一模一样的** `server.New() + Listen("127.0.0.1:0")`，拿到端口后把 Wails 主窗口的 URL 设到 `http://127.0.0.1:<port>/`。

```
┌──────────────── easy-ffmpeg-desktop (Wails) ─────────────────┐
│                                                              │
│  cmd/desktop/main.go                                         │
│    ├─ wails.Run(&options.App{                                │
│    │     Title: "Easy FFmpeg",                               │
│    │     OnStartup: startBackend,  ← 内部启动 server.New     │
│    │     URL: "http://127.0.0.1:<port>/",                    │
│    │  })                                                     │
│    └─ OnShutdown: srv.RequestShutdown()                      │
│                                                              │
│  共享的后端代码（与 Web 版完全相同）：                          │
│    server/  service/  editor/  internal/{job,embedded,...}   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**优点**：

- 前端零改动。SSE、`/api/fs/*`、`/api/editor/*`、剪辑器整套照旧
- 后端零改动。`server.go` 不需要任何 `if wails { ... }` 分支
- 风险面最小：Wails 出 bug 也不影响 Web 版
- 测试可达性最高：两个产物跑的是同一份后端，测一处覆盖两处

**缺点**：

- 仍然占用一个本地端口（127.0.0.1，外部不可访问，无安全损失，但端口冲突理论上存在）
- 没有完全用上 Wails 的"原生 binding"（即不通过 HTTP，直接 JS ↔ Go 调用）。但反过来想，**这正是为什么前端无需重写**

### 方案 B：把后端方法 `Bind` 给 Wails，前端通过 `window.go.*` 调用（不推荐）

Wails 推荐的"原生模式"。前端废弃所有 `fetch(...)` 与 `EventSource(...)`，改用 `window.go.App.ConvertStart(...)` 这类绑定调用，进度通过 `runtime.EventsEmit` 推送。

**为什么不推荐**：

- 前端要重写一整套通信层（`app.js` 的 fetch / `JobBus` 的 EventSource / `EditorApi` / 三个 Tab 的 dryRun + overwrite 流程）
- Web 版必须保留 fetch / SSE，等于要**同时维护两套通信路径**——和"双产物共用代码"的目标背道而驰
- Wails binding 没有 SSE 等价物，要用 `EventsEmit/On` 自己重构进度广播；现有 `internal/job` 的 `Subscribe()` 模型适配成本不低
- 目标群体是"偶尔做格式转换的普通用户"（[overview.md §2](overview.md)），不需要原生通信带来的微秒级延迟优化

### 方案 C：完全不要 Wails，自己拉 WebView（拒绝）

例如直接调 `webview/webview` C 库或系统 API。比 Wails 多花精力实现窗口生命周期、菜单、dialog、信号处理，没收益。

**最终方案**：A。

## 4. 推荐方案落地：双产物的代码组织

### 4.1 新增入口

```
cmd/
├── main.go                    保留：Web 版入口（CGO_ENABLED=0 纯 Go）
├── icon.ico                   保留
├── rsrc_windows.syso          保留（Web 版 .exe 的 Windows 资源）
└── desktop/                   【新增】Wails 桌面版入口
    ├── main.go                wails.Run + OnStartup 调 server.New
    ├── app.go                 *App 结构体（Wails 要求）+ 生命周期钩子
    └── frontend/              Wails 项目结构要求的占位目录
        ├── dist/              空目录（实际 UI 由 server 的 go:embed 提供）
        └── wails.json         Wails 配置（指向上面 dist/）
```

`cmd/main.go` 不动一行代码。`cmd/desktop/main.go` 的骨架：

```go
package main

import (
    "context"
    "easy-ffmpeg/server"
    "easy-ffmpeg/service"
    "log"

    "github.com/wailsapp/wails/v2"
    "github.com/wailsapp/wails/v2/pkg/options"
)

type App struct {
    ctx context.Context
    srv *server.Server
    url string
}

func (a *App) startup(ctx context.Context) {
    a.ctx = ctx
    a.srv = server.New()
    bound, err := a.srv.Listen("127.0.0.1:0")
    if err != nil { log.Fatalf("listen: %v", err) }
    a.url = "http://" + bound + "/"
    go func() { _ = service.Prepare() }()  // 与 Web 版一致：异步解压
}

func (a *App) shutdown(ctx context.Context) {
    a.srv.RequestShutdown()
}

func main() {
    app := &App{}
    err := wails.Run(&options.App{
        Title:     "Easy FFmpeg",
        Width:     1280,
        Height:    800,
        OnStartup: app.startup,
        OnShutdown: app.shutdown,
        // Wails v2 不直接支持"窗口启动后 navigate 到任意 URL"，
        // 实际做法：让 frontend/dist/index.html 作为 shell，启动后用 JS 跳转到 a.url
        // 或直接用 webview 选项把 URL 设进去（具体 API 视 Wails 版本）
    })
    if err != nil { log.Fatal(err) }
}
```

> ⚠️ 上面"navigate 到 HTTP URL"的具体写法需在实施时按 Wails v2 当前版本核对。如果该版本不支持直接指向外部 URL，回退到"frontend/dist/index.html 里写一段 `location.replace('http://127.0.0.1:'+port+'/')`"——端口通过 `runtime.EventsOn` 接收。这是少量且可控的额外胶水代码，不影响主结论。

### 4.2 构建脚本扩展

`build.sh` / `build.bat` 改为产出**两组**产物，桌面版是新增项：

```
dist/
├── easy-ffmpeg.exe                       Web 版 Windows         (CGO=0, 跨编)
├── easy-ffmpeg-macos-arm64               Web 版 macOS arm64     (CGO=0, 跨编)
├── easy-ffmpeg-macos-amd64               Web 版 macOS amd64     (CGO=0, 跨编)
├── easy-ffmpeg-linux                     Web 版 Linux           (CGO=0, 跨编)
├── easy-ffmpeg-desktop.exe               桌面版 Windows         (CGO=1, 本机编)
├── easy-ffmpeg-desktop-macos-arm64.app   桌面版 macOS arm64     (CGO=1, 本机编)
├── easy-ffmpeg-desktop-macos-amd64.app   桌面版 macOS amd64     (CGO=1, 本机编)
└── easy-ffmpeg-desktop-linux             桌面版 Linux           (CGO=1, 本机编)
```

桌面版构建用 `wails build`（背后仍是 `go build` + 资源打包），不能跨平台编译——必须在对应 OS 上跑。脚本应当：

- `build.bat` 在 Windows 上：跑 4 个 Web 版（跨编） + 1 个 Windows 桌面版（本机）
- `build.sh` 在 macOS 上：跑 4 个 Web 版（跨编） + 2 个 macOS 桌面版（arm64 / amd64，本机）
- `build.sh` 在 Linux 上：跑 4 个 Web 版（跨编） + 1 个 Linux 桌面版（本机）

桌面版"全平台齐活"的发布只能等 CI（详见 §7）。

### 4.3 共享 vs 分支的代码点

`go:embed` 嵌入的 `web/` 静态资源和 `internal/embedded/<os>.7z` 都被两个 main 复用——`cmd/main.go` 与 `cmd/desktop/main.go` 都 import `server`，由 Go 链接器负责把对应资源放进各自的二进制。

| 文件 | Web 入口 | 桌面入口 | 备注 |
|------|---------|---------|------|
| `internal/browser/open.go` | 用于打开 UI | 用于"打开输出目录"等 reveal 场景 | 两个入口都 import |
| `cmd/main.go` 里的 `browser.Open(url)` | ✓ | ✗ | 桌面版不需要；Wails 入口里不调 |
| `POST /api/quit` | 唯一退出方式 | 仍可用，但实际由窗口关闭触发 | 不冲突 |
| 控制台进度条（`embedded.startProgressPrinter`） | ✓ | ⚠️ Wails 默认无控制台；进度仍走 `/api/prepare/status` 给前端，不损失体验 | 可选地给 Wails 入口加 `-H windowsgui` 隐藏控制台 |

## 5. 工作拆解

按推荐方案 A 列出实施切片，按依赖顺序排列。每一项后续会作为单独的迭代任务推进。

| # | 切片 | 关键产出 | 备注 |
| --- | ---- | ------- | ---- |
| 0 | 准备 Wails 工具链 | 本机能 `wails build` 成功跑通 hello-world | Windows 装 MinGW-w64；WebView2 Runtime 一般 Win10+ 自带；macOS 装 Xcode CLT；Linux 装 `libwebkit2gtk` 开发包 |
| 1 | `cmd/desktop/main.go` 骨架 | App 结构体 + startup/shutdown 钩子 + `server.New + Listen` | 不动现有 `cmd/main.go` |
| 2 | WebView 指向本地 HTTP URL | 桌面窗口能加载到 Web 版同款 UI | Wails v2 的 URL 设置方式按当时版本核对；最坏回退到 `frontend/dist/index.html` 里 JS 跳转 |
| 3 | 构建脚本扩展 | `build.bat` / `build.sh` 在跑 Web 版 4 产物的基础上追加桌面版本机产物 | 桌面版只能本机编译，按运行平台分支 |
| 4 | Windows 桌面版冒烟 | 双击启动 → 解压进度 → convert / audio / editor 跑通 → 退出 | 主要风险点：navigate URL、SSE 在 WebView2 中表现 |
| 5 | 剪辑器在 WebView 中验证 | `<video>` Range / WebAudio GainNode / gap clock 行为与 Chrome 一致 | 三个浏览器引擎都要过一遍 |
| 6 | macOS 桌面版联调 | `.app` 产物启动 + 关键路径走通 | WKWebView 行为差异；公证推迟到 v0.4.x |
| 7 | Linux 桌面版联调 | 至少 Ubuntu LTS 上跑通 | WebKitGTK 兼容性；不同发行版差异 |
| 8 | 文档同步 | overview.md / roadmap.md / build-and-deploy.md / architecture.md 一致 | 见 §8 列表 |

切片 0–4 是 Windows 桌面版可用的最小集；5 是剪辑器可信度保证；6–7 拓展到三平台；8 收尾。CI 自动化与签名/公证不在本版本（见 §7）。

## 6. 风险与权衡

| 风险 | 严重度 | 缓解 |
|------|--------|------|
| **CGO 强制开启**：Wails 桌面版不能从 Windows 跨编到 macOS/Linux | 中 | 桌面版只能本机编译；通过 GitHub Actions 矩阵解决（§7）。Web 版仍保持跨编不变 |
| **Linux 上 WebKitGTK 是运行时依赖** | 中 | 在 README 里明示需要 `libwebkit2gtk-4.0` / `4.1`；旧发行版用户会受阻 |
| **WebView2 在 Win7/Win8 不可用** | 低 | 项目目标平台已经默认 Win10+；老用户回退到 Web 版 |
| **WebView2 / WKWebView / WebKitGTK 行为不一致** | 中 | 影响最大的是剪辑器（`<video>` / WebAudio / Range 请求）；用方案 A 共享后端的好处是只需排查渲染端，不需要排查 binding 层 |
| **Wails v2 API 演进** | 低 | 官方稳定版迭代不快；锁定 `go.mod` 版本即可 |
| **桌面版二进制变大** | 低 | Windows 上 Wails 不打包 Chromium，只用系统 WebView2，增量约 5–15 MB；总体仍在合理范围 |
| **维护成本翻倍** | 中 | 用方案 A 把"翻倍"压到只是构建/打包/打开方式；运行时逻辑仍是一份 |
| **同一时刻可能跑两个实例（Web + 桌面）** | 低 | 两个进程各自 listen 不同端口，不冲突；但用户可能会困惑。文档说明即可 |

## 7. 工程化（推迟到 v0.4.x）

桌面版必须本机编译这一约束让 CI 变得"必要"而非"可选"。建议在 v0.4.0 主体功能稳定后单独切片：

- **GitHub Actions 矩阵**：windows-latest / macos-13 / macos-14 / ubuntu-latest 四 runner，分别产出 Web 4 个 + 桌面 1–2 个
- **macOS 公证**：申请 Apple Developer 账号或先用 ad-hoc 签名 + `xattr -d com.apple.quarantine` 的引导
- **Windows 代码签名**：暂可缓；SmartScreen 警告与 Web 版一致
- **AppImage 打包**：Linux 桌面版要用户预装 WebKitGTK 是真的折磨；AppImage 打包能缓解但增加复杂度

## 8. 与既有"非目标"的协调

[overview.md §6](overview.md) 与 [roadmap.md §6](roadmap.md) 都明文写过：

> 不打包 webview（不走 Electron / Tauri / Wails 路线）

这是 0.2.x 时代为了"零依赖、可跨编、二进制小"做的决策。本版本撤销该非目标，理由：

1. **目标用户拓展**：Jupyter 式启动对开发者很自然，但部分普通用户被"为啥要打开浏览器"困惑；桌面版降低这个心智门槛
2. **代价已变低**：剪辑器 / 音频 / 转换全部已经成熟，不再有"我们到底要不要做这块功能"的不确定性。引入 Wails 的"重做整个架构"风险已不存在
3. **保留逃生通道**：方案 A 的本质是"加一个外壳"，Web 版完全不退场。用户对桌面版不满意可以回到 Web 版，零迁移成本

需要做的文档同步（v0.4.0 实施时一起改）：

- [overview.md §6](overview.md)：把"不打包 webview"改为"Web 版仍是核心；桌面版为可选 Wails 外壳，详见 v0.4.0"
- [roadmap.md §6](roadmap.md)：同上；并把 Wails 移出"非目标"列表
- [build-and-deploy.md](build-and-deploy.md)：新增"桌面版构建"章节，列 CGO / 工具链要求
- [architecture.md](architecture.md)：在 §1 分层架构图旁加一句"`cmd/main.go` 与 `cmd/desktop/main.go` 是两个并列入口，共享 `server/` 及以下的全部模块"
