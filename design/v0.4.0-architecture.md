## 1. 设计原则（不变量）

落地切片中任何改动都必须满足这五条，违反即否决。

1. **后端零分支**：`server/` 及其下游不得出现 `if wails {}` / `// +build wails` 这类宿主感知代码。Web 版与桌面版跑的必须是**完全相同**的 `server.New() + Listen()` 字节。
2. **前端零改动**：`server/web/`（含 `editor/`）继续用 `fetch` + `EventSource` 与 `http://127.0.0.1:<port>/api/*` 对话；不引入 `window.go.*` 或 `runtime.EventsOn` 等 Wails 原生 binding。
3. **入口并列、不互相依赖**：`cmd/main.go`（Web）与 `cmd/desktop/main.go`（桌面）是 Go 链接器视角下的两个独立 main 包，各自从 `server` import 一份代码副本。Web 入口不感知 Wails 的存在。
4. **桌面版回退路径恒在**：用户对桌面版任何不满都可以回退到 Web 版,无需迁移配置/工程文件（两者共享 `~/.easy-ffmpeg/`）。
5. **跨平台编译保护**：Web 版**继续保持** `CGO_ENABLED=0` 跨编 4 平台。Wails 引入的 CGO 强制开启**只影响桌面版**，必须用 build tag 或独立目录把 cgo 依赖隔离在 `cmd/desktop/` 之内。

## 2. 高层架构

```
                     ┌────────────────────────────────────────────┐
                     │       共享层（Web 版与桌面版完全相同）         │
                     │  server/      service/      editor/         │
                     │  internal/{job,embedded,browser,procutil}   │
                     │  config/                                    │
                     │  go:embed web/  +  embedded/<os>/*.7z       │
                     └─────────────────────┬──────────────────────┘
                                           │
                  ┌────────────────────────┼────────────────────────┐
                  │                        │                        │
                  ▼                        ▼                        ▼
         ┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐
         │  cmd/main.go    │      │ cmd/desktop/    │      │ （未来）         │
         │  Web 版入口      │      │  main.go        │      │ cmd/easy-editor │
         │                 │      │  桌面版入口       │      │  剪辑器独立 exe  │
         │ server.Listen   │      │ wails.Run        │      │                 │
         │ + browser.Open  │      │  + server.Listen │      │                 │
         │ + Ctrl+C 监听    │      │  + WebView 指向 │      │                 │
         └─────────────────┘      │    localhost:port│      └─────────────────┘
                                  └─────────────────┘
                                           │
                                           ▼
                                  ┌─────────────────┐
                                  │  WebView2  /    │
                                  │  WKWebView /    │
                                  │  WebKitGTK      │
                                  │  （平台原生）     │
                                  └─────────────────┘
```

**核心拓扑**：桌面版进程内同时运行
- 一个 Wails 主窗口（持有平台 WebView）
- 一个 `127.0.0.1:<random>` 上的 HTTP 服务（与 Web 版同构）
- WebView 的 `URL` 指向该服务

通信路径**仍是 HTTP/SSE**，不走 Wails binding。这是方案 A 的全部精髓。

**为什么把 Wails 配置放 `cmd/desktop/` 而不是仓库根**：

- Wails 默认假设 `wails.json` 在仓库根并把它当作整个项目根目录。我们反过来：让 `cmd/desktop/` 自成一个 Wails 子项目，仓库根继续是 Go module 根。
- 这样 `wails build` 的工作目录就是 `cmd/desktop/`，不污染仓库其他部分；同时 Go module 边界（`go.mod` 在仓库根）保持不变，`server/` 等共享包通过 `easy-ffmpeg/server` 路径正常 import。
- `cmd/desktop/build/` 和 `cmd/desktop/frontend/` 等 Wails 工具链产物全部进 `.gitignore`。

## 4. 桌面版入口详细设计

### 4.1 `cmd/desktop/main.go` 与 `app.go`

职责切分：
- `main.go` 只做 `wails.Run` 调用与 `App` 构造；
- `app.go` 持有所有运行时状态（`*server.Server`、bound URL）和生命周期钩子。

```go
// cmd/desktop/app.go
package main

import (
    "context"
    "log"

    "easy-ffmpeg/server"
    "easy-ffmpeg/service"

    "github.com/wailsapp/wails/v2/pkg/runtime"
)

// App 是 Wails 应用的状态容器，**不是**业务逻辑承载者。所有业务都在
// 共享的 server.Server 里跑；这里只负责"何时启动 / 何时停止"。
type App struct {
    ctx context.Context
    srv *server.Server
    url string // 例如 "http://127.0.0.1:54321/"
}

// startup 在 Wails 主窗口创建后、首帧渲染前调用（Wails OnStartup 契约）。
// 1) 启动与 Web 版完全相同的 HTTP 服务于 127.0.0.1:0；
// 2) 后台异步触发 FFmpeg 解压（与 Web 版一致）；
// 3) 把 bound URL 用 runtime.EventsEmit 推给前端 shell（fallback 路径用）；
//    或在 §4.2 主路径下，由 main.go 在 wails.Run 选项里直接把 URL 设为 a.url。
func (a *App) startup(ctx context.Context) {
    a.ctx = ctx
    a.srv = server.New()
    bound, err := a.srv.Listen("127.0.0.1:0")
    if err != nil {
        log.Fatalf("desktop: listen failed: %v", err)
    }
    a.url = "http://" + bound + "/"
    log.Printf("desktop: backend bound at %s", a.url)

    go func() {
        if err := service.Prepare(); err != nil {
            log.Printf("desktop: ffmpeg prepare failed: %v", err)
        }
    }()

    runtime.EventsEmit(ctx, "backend-ready", a.url)
}

// shutdown 在用户关窗时被 Wails 调用。我们 reuse Web 版同一条优雅关停
// 路径：通知 server.Wait 解除阻塞，HTTP server 收到 Shutdown 信号。
func (a *App) shutdown(ctx context.Context) {
    if a.srv != nil {
        a.srv.RequestShutdown()
    }
}
```

```go
// cmd/desktop/main.go
package main

import (
    "embed"
    "log"

    "github.com/wailsapp/wails/v2"
    "github.com/wailsapp/wails/v2/pkg/options"
    "github.com/wailsapp/wails/v2/pkg/options/assetserver"
)

//go:embed all:frontend/dist
var assets embed.FS // shell.html 所在目录；fallback 路径用

func main() {
    app := &App{}
    err := wails.Run(&options.App{
        Title:      "Easy FFmpeg",
        Width:      1280,
        Height:     800,
        MinWidth:   900,
        MinHeight:  600,
        OnStartup:  app.startup,
        OnShutdown: app.shutdown,
        AssetServer: &assetserver.Options{
            Assets: assets,
        },
        // 启动时加载的初始页面：见 §4.2 的两条路径。
    })
    if err != nil {
        log.Fatalf("wails.Run: %v", err)
    }
}
```

### 4.2 WebView 加载本地 HTTP URL：主路径 + 回退路径

这是方案 A 唯一一处真正需要"按 Wails 版本核对"的胶水。设计上准备**两条路径**，主路径不行就走回退路径，对前端代码完全透明。

#### 主路径 P1：让 Wails 直接 navigate 到 `http://127.0.0.1:<port>/`

如果当前 Wails v2 版本支持在 `OnDomReady` 或类似钩子中调用 `runtime.WindowReload(ctx)` 或直接设定外部 URL，则 `App.startup` 拿到端口后立刻把窗口导航过去。前端是真实的 `server/web/index.html`，与 Web 版完全相同。

#### 回退路径 P2：shell.html + JS `location.replace`

若 P1 不可用（Wails AssetServer 不允许指向外部 origin），则：

1. `cmd/desktop/frontend/dist/index.html` 是一个**极简 shell**：
    ```html
    <!doctype html>
    <html><head><meta charset="utf-8"><title>Easy FFmpeg</title></head>
    <body>
      <script>
        // 监听后端 ready 事件，拿到 URL 后整页跳转
        window.runtime.EventsOn('backend-ready', (url) => {
            location.replace(url);
        });
      </script>
    </body></html>
    ```
2. `App.startup` 通过 `runtime.EventsEmit(ctx, "backend-ready", a.url)` 推送端口；
3. 跳转后前端进入正常的 `server/web/` 资源，与 Web 版**字节相同**。

> ⚠️ 切片 2 的验收：在 Windows WebView2 上确认主路径 P1 是否可用；不可用即落回退 P2 并把它固化为唯一实现。文档同步更新此节。

**两条路径对前端的等价性证明**：跳转后 `document.URL` 都是 `http://127.0.0.1:<port>/`，所有 `fetch('/api/...')` / `new EventSource('/api/...')` 的相对路径都解析到同一个 origin。前端代码无任何条件分支。

### 4.3 生命周期映射

| 事件 | Web 版 | 桌面版 |
|------|--------|--------|
| 启动 | `main.go` 同步 `Listen` + 异步 `Prepare` + `browser.Open` + `Wait` | `wails.Run` → `OnStartup` 调 `Listen` + `Prepare`；不调 `browser.Open` |
| 解压进度 | 控制台进度条 + 浏览器轮询 `/api/prepare/status` | **只有**浏览器轮询（无控制台）→ shell 加载完成后看到与 Web 版同款遮罩进度 |
| 退出 | `Ctrl+C` 或 `POST /api/quit` 触发 `srv.RequestShutdown` | 关窗触发 `OnShutdown` → `srv.RequestShutdown`；`POST /api/quit` 仍可用且无害 |
| 控制台 | 有（启动信息 + 进度条 + 日志） | 无（Wails 默认不附控制台；Windows 上需 `-H windowsgui` 等价的 Wails 选项） |

**已知行为差异**：

- **解压进度**桌面版用户**只在窗口里看到**遮罩进度，看不到 Web 版的控制台进度条。这是预期行为，不是 bug；底层 `embedded.startProgressPrinter` 在桌面版也启动但写到 `os.Stderr` 是无人看的（Windows GUI 应用无控制台）。后续切片可考虑桌面版**不启动** printer goroutine 节省一点资源；非必需。
- **`browser.Open` 的另一处用途仍保留**：在 reveal 文件夹（`/api/fs/reveal`）等场景下，桌面版仍 import [internal/browser/open.go](../internal/browser/open.go) 调用系统文件管理器。

### 4.4 端口冲突与单实例

桌面版与 Web 版**不互斥**——它们各自 `Listen("127.0.0.1:0")` 拿随机端口，可以共存。文档需说明：

- 同时跑 Web 版和桌面版会**有两个独立的进程、两个端口、共享 `~/.easy-ffmpeg/`**；解压缓存复用，但运行任务彼此不可见（`internal/job` 是进程内单例）。
- 用户大多不会主动这么做；不强制单实例锁。

## 5. 后端共享层的"不可逾越的边界"

这一节列出**所有** Web 版与桌面版必须保持完全一致的代码点，作为代码评审的对照清单。

### 5.1 `server/server.go`

- `New() *Server`、`Listen(addr) (string, error)`、`Wait()`、`RequestShutdown()` 的签名与语义**完全冻结**。
- `routes(mux)` 不得新增宿主感知分支。
- `silentPaths` 列表不得为桌面版新增条目（除非两者都要静默）。
- `logMiddleware` 不得感知 Wails。

### 5.2 `server/web/`

- `index.html` 不得引入任何 `if (window.runtime) {}` 之类的桌面端探测分支。整套前端**只知道自己在浏览器里**——是 Chrome、WebView2 还是 WKWebView 对它透明。
- 新增任何"桌面专属"按钮（如"最小化到托盘"）一律推迟到 v0.4.x 之后，并通过**新的后端 API + 服务端 feature gate** 提供，不在前端做 UA sniff。

### 5.3 `editor/`

- 模块入口 `editor.NewModule(Deps)` 已有的 SOLID 设计天然支持双产物；桌面版的 `cmd/desktop/` 不直接 import `editor`，而是经由 `server.New()` 间接装配（与 Web 版同构）。
- `editor/api/handlers_source.go` 的 HTTP Range 实现是剪辑器在 WebView 中能否正常预览的关键，**不得改**。

### 5.4 `internal/embedded/`

- `service.Prepare()` 路径不变；`~/.easy-ffmpeg/bin-<hash>/` 缓存被两个产物共享。
- `Progress` 数据结构与 `/api/prepare/status` 端点不变。

### 5.5 `internal/job/`、`internal/browser/`、`internal/procutil/`

- 全部不动。`procutil.HideWindow`（CREATE_NO_WINDOW）在桌面版下行为相同——它隐藏的是 ffmpeg 子进程窗口，与外层是浏览器还是 Wails 无关。

## 6. 跨平台编译策略

### 6.1 cgo 隔离

**核心约束**：Wails 强制 `CGO_ENABLED=1` 且依赖平台 C 工具链；Web 版必须保持 `CGO_ENABLED=0` 跨编。

实现手段：**入口隔离 + 依赖隔离**。

- 所有 Wails 相关 import (`github.com/wailsapp/wails/v2/...`) **只**出现在 `cmd/desktop/` 包内。
- `server/`、`editor/`、`internal/`、`service/`、`config/` 等共享包**严禁** import Wails。
- 这样 `go build ./cmd`（Web 版）即使在 `CGO_ENABLED=0` 下也能成功，因为依赖图根本不含 cgo 符号。
- `go build ./cmd/desktop` 才会触发 cgo 编译，此时必须 `CGO_ENABLED=1` + 本机 C 工具链。

**验收**：CI（v0.4.x）上加一条 `CGO_ENABLED=0 go build ./cmd` 检查，防止误把 Wails import 渗入共享层。

### 6.2 构建矩阵

| 产物 | CGO | 编译方式 | 可在哪台机器编 |
|------|-----|----------|----------------|
| `easy-ffmpeg.exe`（Web Win） | 0 | 跨编 | Win / macOS / Linux 都可 |
| `easy-ffmpeg-macos-arm64`（Web） | 0 | 跨编 | Win / macOS / Linux 都可 |
| `easy-ffmpeg-macos-amd64`（Web） | 0 | 跨编 | Win / macOS / Linux 都可 |
| `easy-ffmpeg-linux`（Web） | 0 | 跨编 | Win / macOS / Linux 都可 |
| `easy-ffmpeg-desktop.exe`（桌面 Win） | 1 | `wails build` | **仅** Windows + MinGW-w64 |
| `easy-ffmpeg-desktop-macos-arm64.app` | 1 | `wails build -platform darwin/arm64` | **仅** macOS arm64（或 Apple Silicon 上交叉到 amd64） |
| `easy-ffmpeg-desktop-macos-amd64.app` | 1 | `wails build -platform darwin/amd64` | **仅** macOS（同上） |
| `easy-ffmpeg-desktop-linux`（桌面 Linux） | 1 | `wails build` | **仅** Linux + libwebkit2gtk-dev |

### 6.3 `build.bat` / `build.sh` 扩展

两个脚本保留现有 Web 4 产物逻辑不变，**追加**桌面版分支：

#### `build.sh`（macOS / Linux / Windows Git Bash）

```bash
# ... Web 4 产物保持不变 ...

# Desktop section: 仅在 wails CLI 可用且当前是桌面版可编译的平台时执行
build_desktop() {
    if ! command -v wails >/dev/null 2>&1; then
        echo "==> wails CLI not found, skipping desktop build"
        return
    fi
    case "$(uname -s)" in
        Darwin)
            (cd cmd/desktop && wails build -platform darwin/arm64 -o ../../dist/easy-ffmpeg-desktop-macos-arm64.app)
            (cd cmd/desktop && wails build -platform darwin/amd64 -o ../../dist/easy-ffmpeg-desktop-macos-amd64.app)
            ;;
        Linux)
            (cd cmd/desktop && wails build -o ../../dist/easy-ffmpeg-desktop-linux)
            ;;
        MINGW*|MSYS*|CYGWIN*)
            (cd cmd/desktop && wails build -o ../../dist/easy-ffmpeg-desktop.exe)
            ;;
    esac
}

build_desktop
```

#### `build.bat`（Windows）

```batch
REM ... Web 4 产物保持不变 ...

where wails >nul 2>&1
if errorlevel 1 (
    echo ==^> wails CLI not found, skipping desktop build
    goto :done
)

pushd cmd\desktop
wails build -o ..\..\dist\easy-ffmpeg-desktop.exe || (popd & goto :fail)
popd

:done
```

**关键设计**：
- 脚本**自动跳过**当前机器编不了的桌面产物，不报错。这样开发者本机一键 `build.bat` 仍能拿到全部 Web 产物 + 一个桌面产物，CI 矩阵则覆盖所有 4 个桌面产物。
- 桌面版编译**前提**是 7z 文件已就位（同 Web 版），因为 `cmd/desktop/main.go` 通过 `import "easy-ffmpeg/server"` 把同一份 `go:embed` 拉进二进制。

## 7. 平台 runtime 依赖

| 平台 | WebView 实现 | 用户依赖 | 处理 |
|------|--------------|----------|------|
| Windows 10+ | WebView2 | Runtime 通常预装；个别老镜像没有 | README 写明；Wails 安装包可选地引导用户装 Evergreen Runtime |
| Windows 7/8 | — | WebView2 不支持 | **不支持桌面版**；引导回退到 Web 版 |
| macOS 11+ | WKWebView | 系统自带，零依赖 | — |
| Linux | WebKitGTK | `libwebkit2gtk-4.0-37` 或 `4.1-0` | README 列出 apt/dnf 包名；旧发行版（CentOS 7）受阻 |

`internal/embedded/<os>.7z` 与平台 runtime 解耦——它只决定"程序能跑 ffmpeg"，与"程序能开窗"是两件事。

## 8. WebView 引擎差异点（剪辑器关注）

剪辑器是三大 Tab 中**唯一**深度依赖浏览器多媒体能力的模块。三引擎差异点清单（切片 5 的验收依据）：

| 能力 | Chrome（Web 版） | WebView2（Win） | WKWebView（macOS） | WebKitGTK（Linux） |
|------|----------------|----------------|--------------------|--------------------|
| `<video>` HTTP Range 请求 | 标配 | 同 Chromium | 通常 OK，少数 codec 走不同路径 | OK |
| `<video>` muted 自动播放 | OK | OK | 严格策略（页面需用户手势） | OK |
| WebAudio `GainNode` 音量调节 | OK | OK | OK | OK |
| `MediaSource` / SourceBuffer | OK | OK | 部分 codec 受限 | 受 GStreamer 插件影响 |
| 拖拽 + DataTransfer | OK | OK | OK | OK |
| `EventSource`（SSE） | OK | OK | OK | OK |
| 编码兼容（h265/hevc）| 视系统 | 视系统 | macOS 12+ OK | 受 GStreamer 影响，常缺 |

> 9d92bba 提交"剪辑针对浏览器无法播放的编码进行提示"已经在 Web 版处理了一部分，桌面版直接复用同一段 JS，但**实测覆盖范围**会因 WebView 引擎不同而扩大。

**设计上的对策**：
- 不在前端做 `navigator.userAgent` 分支；
- "无法预览" 走与 Web 版相同的提示路径，按 codec 而非按引擎判定；
- 切片 5 的验收必须**在三个引擎上各跑一次**整套剪辑流程（导入 → split → 调音量 → 导出）。

## 9. 启动时序（桌面版）

对照 [architecture.md §3](architecture.md) 的 Web 版时序：

```
T=0.0s   wails.Run 启动 → 创建主窗口（暂未加载内容）
T=0.0s   OnStartup 钩子触发
         ├─ a.srv = server.New()
         ├─ a.srv.Listen("127.0.0.1:0") → bound = "127.0.0.1:54321"
         ├─ a.url = "http://127.0.0.1:54321/"
         └─ go service.Prepare() 异步启动解压
T=0.05s  WebView 完成初始 navigate
         · P1 路径：直接 navigate 到 a.url
         · P2 路径：加载 shell.html → JS 监听 backend-ready 事件 → location.replace
T=0.3s   前端 app.js 启动 → 调 /api/prepare/status
         └─ state="extracting" → 显示遮罩 + 进度条
T=~40s   解压完成 → /api/prepare/status 返回 ready → 遮罩淡出
T=...    用户操作 = 与 Web 版完全相同
T=close  用户关窗
         ├─ Wails 触发 OnShutdown
         ├─ a.srv.RequestShutdown()
         ├─ http.Server.Shutdown(3s timeout)
         └─ 进程退出
```

**与 Web 版的唯一实质性差异**：T=0.05s 的 navigate 来源不同（一个是浏览器自己 `browser.Open`，一个是 Wails WebView 内部）。从 T=0.3s 之后两者完全同构。

## 11. 测试策略

### 11.1 自动化测试（不变 + 新增）

- 现有 `server/audio_args_test.go`、`editor/domain/*_test.go`、`editor/storage/jsonrepo_test.go` **全部跑过两次**：一次普通 `go test`，一次 `CGO_ENABLED=0 go test`。后者保证共享层未渗入 cgo。
- **新增**`cmd/desktop/` 下不写单元测试。Wails 入口逻辑很薄，经 §11.2 端到端验证即可。

### 11.2 端到端冒烟矩阵（人工，每次发版前过一遍）

按 (产物 × 平台 × 关键路径) 三维矩阵，**每个组合都必须过**：

| 产物 | 平台 | 必过路径 |
|------|------|----------|
| Web | Win10/11 | 解压 → convert → audio extract → editor split + 导出 → 退出 |
| Web | macOS 14 | 同上 |
| Web | Ubuntu 22.04 | 同上 |
| 桌面 | Win10/11 | 解压（窗口内进度）→ convert → audio extract → editor split + 导出 → 关窗即退 |

| 桌面 | Ubuntu 22.04 | 同上，含 libwebkit2gtk 缺失时的降级提示 |

剪辑器路径单独再加一项：在三个 WebView 引擎里都验证 §8 列出的所有能力点。

### 11.3 回归保护

v0.3.x 已发布的所有功能，在 Web 版三平台上必须**零行为变化**（包括日志格式、SSE 字段、JSON 工程兼容）。这是不变量 #1 的可观测体现。
